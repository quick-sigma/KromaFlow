# S010 — Step Abstraction Layer

## Resumen

Se creó una nueva capa de abstracción `Step` que envuelve `Processor` y `OutputFormatter` como nodos de pipeline descubribles por el frontend. Cada Step expone metadatos (name, description, version), un esquema de configuración Pydantic (en formato JSON Schema), y un `@register` decorator para registro automático.

Se agregó el endpoint `GET /api/steps` que lista todos los steps registrados con su metadata.

### Archivos creados

#### `Backend/step.py` — Core abstraction

| Componente | Descripción |
|---|---|
| `StepVariant` | Enum: `PROCESSOR` / `OUTPUT_FORMATTER` |
| `StepInfo` | Pydantic model para comunicación con frontend |
| `Step[ConfigT]` | Clase genérica que wrappea un componente con metadata |
| `register()` | Decorator que puebla el `_registry` global |
| `get_registered_steps()` | Retorna `list[StepInfo]` para API |

El método `Step.execute()` despacha según el variant:
- `PROCESSOR`: llama `component.process(image, config)`
- `OUTPUT_FORMATTER`: llama `component.format_output(image, output_format, quality)` con kwargs

#### `Backend/steps_config.py` — Config schemas y Steps concretos

| Step | Variant | Config Schema | Componente envuelto |
|---|---|---|---|
| `ImageProcessorStep` | PROCESSOR | `ImageProcessingConfig` (subclase de `ProcessingInstructions`) | `ImageProcessor` |
| `WatermarkRemovalStep` | PROCESSOR | `WatermarkRemovalConfig` (`enabled: bool`) | `WatermarkRemoverProcessor` (optional) |
| `ImageOutputFormatterStep` | OUTPUT_FORMATTER | `OutputFormatConfig` (`format`, `quality`) | `ImageOutputFormatter` |
| `AVIFOutputFormatterStep` | OUTPUT_FORMATTER | `OutputFormatConfig` | `AVIFOutputFormatter` (optional) |

Steps con dependencias opcionales (`watermark-remover`, `avif-output-formatter`) — si falla el import, `get_registered_steps()` los salta silenciosamente.
