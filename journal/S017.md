# S017 — SyncEngine: persistent state across sessions

## Summary

Implemented a `SyncEngine` — an IndexedDB-backed persistence layer that live-syncs all app state (images, pipeline editor, step cache) between sessions. If the window is refreshed, the app restores its exact state: loaded images, pipeline steps, per-step configuration, and cached step metadata.

## Architecture

```
SyncEngine (singleton)
├── IndexedDB database: "image-prepare-sync" (v1)
│   ├── ObjectStore "state"   → Zustand persist payloads (JSON)
│   └── ObjectStore "blobs"   → Image binary data (Blob)
├── In-memory fallback (Map)  → Used when IndexedDB is unavailable (jsdom, private browsing)
├── Debounced batch writes    → 100ms debounce coalesces rapid state changes
└── Zustand storage adapter   → Bridges IndexedDB ↔ Zustand persist middleware
```

## SyncEngine details

### IndexedDB schema

- **Database**: `image-prepare-sync`, version 1
- **Object store `state`**: key-value pairs for Zustand persist (store name → JSON string)
- **Object store `blobs`**: key-value pairs for image binary data (blob key → Blob)

### Fallback behavior

When IndexedDB is unavailable (jsdom, private browsing, or `open()` fails), the SyncEngine transparently falls back to an in-memory `Map`. This ensures tests work without modification and the app degrades gracefully.

### Debounced writes

Instead of writing to IndexedDB on every state change, writes are coalesced within a 100ms window. This prevents excessive IndexedDB transactions during rapid state changes (e.g., slider drag in step config).

```
enqueueStateWrite("pipeline-store", json)
enqueueStateWrite("pipeline-store", json)  ← replaces previous
  └── after 100ms: single IndexedDB transaction writes latest value
```

## Per-store persistence strategy

### Pipeline store (`pipeline-store` key)

- **Persisted**: `steps[]` — the current pipeline step array
- **Hydration**: On page load, PipelineEditor initializes local state from the persisted global store after the hydration gate completes
- **Sync**: Local state in PipelineEditor → global store via existing `useEffect` → persist middleware auto-saves

### Images store (`images-store` key)

- **Persisted (via `partialize`)**: `{ id, name, type, size, blobKey }[]` — metadata only
- **Not persisted**: `file: File` and `src: string` (blob URL) — reconstructed on hydration
- **Blob storage**: Each image's binary data is stored in the `blobs` object store with key `image-blob-{uuid}`
- **Hydration flow**:
  1. Page loads → Zustand persist hydrates image metadata from `state` store
  2. `onRehydrateStorage` callback fires → schedules `hydrateBlobs()`
  3. `hydrateBlobs()` loads each blob from `blobs` store → creates `new File(...)` and `URL.createObjectURL(...)`
  4. Store is updated with fully reconstructed `ImageEntry[]`
- **Cleanup**: `removeImage()` and `clearImages()` delete blobs from IndexedDB
- **Orphan detection**: `cleanupOrphanedBlobs()` runs after hydration to remove blobs without matching image entries

### Steps store (`steps-store` key)

- **Persisted**: `steps[]` and `lastFetchedAt` timestamp
- **TTL**: 5-minute cache. `loadSteps()` returns cached data if TTL hasn't expired
- **Graceful degradation**: If the API call fails but cached data exists, the cache is used silently (no error shown)

## Hydration gate (`App.tsx`)

The `useHydrationGate()` hook:

1. Calls `syncEngine.ready()` to ensure IndexedDB is initialized
2. Calls `rehydrate()` on all persisted stores to force state loading
3. Calls `hydrateBlobs()` to reconstruct image File objects and blob URLs
4. Sets `ready = true` — renders the main UI

While hydrating, a loading spinner with "Restoring session…" text is shown. This prevents components from rendering with incomplete state (e.g., image metadata without blob URLs).

## Edge cases handled

| Case | Handling |
|------|----------|
| IndexedDB unavailable | In-memory Map fallback |
| IndexedDB open fails | Graceful fallback with console warning |
| Private browsing | In-memory Map fallback |
| Blob missing on hydration | Entry skipped, warning logged |
| Partial blob save failure | Cleanup of already-saved blobs |
| Orphaned blobs | Cleanup after hydration |
| API fetch fails with cached data | Silent use of cached steps |
| Corrupted persist state | `try/catch` in `createJSONStorage` returns null → default state |
| Rapid state changes | Debounced writes (100ms) |
| Window refresh | Full state restoration via IndexedDB |

## Future considerations

- **Multi-tab sync**: Use IndexedDB `onversionchange` or `BroadcastChannel` API to sync state across tabs
- **Storage quota**: IndexedDB has generous limits (~50% of disk), but individual large images could add up. Consider compression or user-facing quota management.
- **Backend sync**: For multi-device scenarios, a server-side persistence layer could be added behind the SyncEngine abstraction
